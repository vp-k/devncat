# Post-Analysis: Metrics Recommendation

소스 코드를 분석하여 측정해야 할 KPI와 Analytics 이벤트를 추천합니다.

## 입력

- 소스 코드 (lib/, src/, app/ 등)
- 오케스트레이터가 수집한 공통 맥락

## 분석 절차

### Step 1: 기능 추출

소스 코드에서 사용자 대면 기능을 식별합니다:

1. **라우트/화면 목록**: 라우터 설정 파일, 페이지 컴포넌트, 네비게이션 정의에서 추출
2. **API 엔드포인트**: REST/GraphQL 엔드포인트, RPC 함수 목록
3. **사용자 액션**: 폼 제출, 버튼 클릭, 상태 변경 등 인터랙션 포인트

각 기능에 대해 한 줄 설명을 작성합니다.

### Step 2: 기능별 KPI 매핑

각 기능에 3~5개 KPI를 매핑합니다. 다음 카테고리에서 선택:

| 카테고리 | 예시 지표 |
|---------|----------|
| 사용률 | DAU, 기능별 사용 비율, 세션당 사용 횟수 |
| 전환율 | 퍼널 단계별 전환, 가입→활성화, 무료→유료 |
| 참여도 | 세션 길이, 재방문율, 기능 깊이 |
| 이탈률 | 화면별 이탈, 에러 후 이탈, 퍼널 이탈 |
| 에러율 | API 실패율, 크래시율, 타임아웃 비율 |

### Step 3: Analytics 이벤트 코드 제안

프로젝트에서 사용 중인 Analytics 라이브러리를 감지합니다:

- **Flutter**: firebase_analytics, mixpanel_flutter, amplitude_flutter, posthog
- **Web**: Google Analytics, Mixpanel, Amplitude, PostHog, Segment
- **Backend**: 커스텀 이벤트 로깅

감지된 라이브러리 기반으로 구체적인 이벤트 추적 코드를 제안합니다.
라이브러리 미감지 시 의사코드(pseudocode)로 제안합니다.

각 이벤트에 포함할 속성:
- 이벤트 이름 (snake_case)
- 필수 속성 (parameters)
- 트리거 조건

### Step 4: 대시보드 구성 초안

1. **NSM (North Star Metric)** 1개 선정 + 근거
2. **핵심 패널 4~6개** 구성:
   - 각 패널: 제목, 포함 지표 2~3개, 시각화 타입 (line/bar/funnel/number)
   - 알림 기준값 (threshold) 제안
3. 데이터 수집 주기 제안 (실시간/시간별/일별)

## 출력

결과를 `docs/post-analysis/metrics-recommendation.md`에 저장합니다.

|---|--------|------|--------|

### [기능명]

|----------|------|------|
| ... | ... | ... |
