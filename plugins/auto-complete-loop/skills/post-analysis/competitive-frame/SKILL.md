# Post-Analysis: Competitive Frame

소스 코드와 기획 문서를 기반으로 경쟁 분석 프레임을 생성합니다.
우리 앱의 기능은 코드에서 자동 추출하고, 경쟁사 정보는 `[조사 필요]`로 표시합니다.

## 입력

- 소스 코드 (라우트, 화면, API 엔드포인트)
- 기획 문서 (docs/ 내 overview.md, SPEC.md 등)
- 오케스트레이터가 수집한 공통 맥락

## 분석 절차

### Step 1: 기능 목록 자동 추출

소스 코드에서 사용자 대면 기능을 체계적으로 추출:

1. **화면/페이지 목록**: 라우터 설정, 페이지 컴포넌트에서 추출
2. **주요 사용자 액션**: CRUD 연산, 폼 제출, 상태 변경
3. **시스템 기능**: 인증, 알림, 검색, 설정 등
4. **통합 기능**: 외부 서비스 연동 (결제, 소셜 로그인, 지도 등)

각 기능을 카테고리별로 분류:
- **Core**: 앱의 핵심 가치를 전달하는 기능
- **Supporting**: 핵심 기능을 보조하는 기능
- **Infrastructure**: 인증, 설정 등 기반 기능

### Step 2: 기능 비교 테이블

추출된 기능을 경쟁사와 비교하는 프레임 생성:

| 카테고리 | 기능 | 우리 앱 | 경쟁앱 A | 경쟁앱 B | 경쟁앱 C |
|---------|------|---------|---------|---------|---------|
| Core | 기능1 | 구현됨 | [조사 필요] | [조사 필요] | [조사 필요] |
| Core | 기능2 | 구현됨 | [조사 필요] | [조사 필요] | [조사 필요] |
| ... | ... | ... | ... | ... | ... |

경쟁앱 이름은 기획 문서에 언급된 경우 사용하고, 없으면 "경쟁앱 A/B/C"로 플레이스홀더 처리합니다.

### Step 3: 차별화 포인트

코드와 문서에서 식별 가능한 강점:

1. **독특한 기능**: 경쟁사 대비 차별화된 기능 (기획 문서 또는 코드에서 고유한 것)
2. **기술적 강점**: 사용된 기술 스택, 아키텍처 패턴, 성능 최적화
3. **UX 강점**: 코드에서 파악 가능한 UX 패턴 (오프라인 지원, 로딩 최적화, 접근성 등)
4. **확장성**: 아키텍처의 확장 가능성 (Clean Architecture, 모듈화 등)

### Step 4: 약점/보완점

카테고리 일반 기능 중 미구현 기능을 식별하고 우선순위를 매깁니다:

1. **카테고리 추론**: 앱의 기능과 기획 문서에서 앱 카테고리 추론
2. **일반 기능 대조**: 해당 카테고리에서 일반적으로 기대되는 기능 목록 생성
3. **Gap 식별**: 일반 기능 중 현재 미구현된 항목
4. **우선순위 분류**:
   - **P1 (Must-have)**: 카테고리 앱에서 사용자가 당연히 기대하는 기능
   - **P2 (Should-have)**: 경쟁력을 위해 필요하지만 당장 없어도 동작하는 기능
   - **P3 (Nice-to-have)**: 차별화에 도움이 되지만 우선순위가 낮은 기능
